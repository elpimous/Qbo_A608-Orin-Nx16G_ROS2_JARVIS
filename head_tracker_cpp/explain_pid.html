1. Principe du PID dans le head tracker

Le PID (Proportionnel, Intégral, Dérivé) est un contrôleur en boucle fermée. Dans le head_tracker_node, il prend en entrée l'erreur angulaire (écart en radians entre le centre de l'image et la cible détectée) et produit une correction de position envoyée au servo Dynamixel via /cmd_joints.

La vitesse du servo est calculée séparément, proportionnelle à l'amplitude de l'erreur : grosse erreur → mouvement rapide, petite erreur → mouvement lent et précis. Un filtre passe-bas lisse le tout pour éviter les saccades.

Boucle complète : caméra → détection (ROI) → erreur pixel → conversion rad → PID → correction position → filtre passe-bas → seuil commande → /cmd_joints → servo Dynamixel → mouvement tête → caméra (retour boucle)

2. Les 3 composantes en détail

P — Proportionnel (force de rappel)

correction_P = Kp × erreur

Plus la cible est loin du centre image, plus la correction est forte. C'est la composante principale — sans elle, rien ne bouge.

Trop haut : la tête dépasse la cible (overshoot) et oscille autour.
Trop bas : la tête met trop longtemps à rejoindre la cible, ou n'y arrive pas.

I — Intégral (correction de biais)

correction_I = Ki × ∑(erreur × dt)

Accumule l'erreur dans le temps. Si la tête reste décalée de 2 pixels pendant 100 frames, l'intégrale grossit et pousse la correction jusqu'à annuler ce biais.

Trop haut : l'accumulation dépasse la cible → oscillations lentes (la tête "dérive").
Trop bas / zéro : un petit décalage permanent peut rester (erreur statique).

L'anti-windup (max_integral = 1.0) empêche l'intégrale d'exploser quand la cible disparaît longtemps.

D — Dérivé (frein / amortisseur)

correction_D = Kd × (erreur - erreur_précédente) / dt

Regarde la vitesse de changement de l'erreur. Quand l'erreur diminue (la tête approche de la cible), D produit une correction négative → il freine.

C'est le paramètre anti-oscillation. C'est celui qui te manquait.

Trop haut : la tête devient "visqueuse", lente, et amplifie le bruit du capteur.
Trop bas : pas de freinage → la tête dépasse la cible à chaque correction.

Le ratio clé : Kd / Kp
Avec tes anciennes valeurs (Kp=1.0, Kd=0.02), le ratio est 0.02 — quasi pas d'amortissement.
Valeur recommandée : ratio entre 0.08 et 0.15 → suffisamment de freinage pour éliminer l'overshoot.

Calcul de la vitesse (séparé du PID)

La vitesse envoyée au servo est indépendante du PID. Elle est proportionnelle à l'erreur : 

vel = clamp(|erreur_rad| × vel_gain, min_vel, max_vel)

Avec vel_gain = 4.0, une erreur de 0.3 rad donne 1.2 rad/s. Le servo arrive vite quand la cible est loin, lentement quand elle est proche. Cette vitesse est ensuite lissée par le même filtre passe-bas que la position.

5. Diagnostic des symptômes

Symptôme observé Cause probable Action
La tête oscille autour de la cible Kp trop haut et/ou Kd trop bas ↓ Kp ↑ Kd
Oscillation lente (la tête "dérive") Ki trop haut → accumulation excessive ↓ Ki ou mettre à 0
La tête met trop longtemps à suivre Kp trop bas ou vel_gain trop bas ↑ Kp ↑ vel_gain
Saccades rapides / vibrations Lissage insuffisant ou seuil trop bas ↓ alpha ↑ deadzone
Petit décalage permanent Pas assez de Ki (erreur statique) ↑ Ki (léger, 0.01-0.03)
La tête est "molle" / visqueuse Kd trop haut → trop de freinage ↓ Kd
Mouvement lent mais sans oscillation vel_gain ou max_vel trop bas ↑ vel_gain ↑ max_vel
Mouvement OK mais bruit quand c'est centré Deadzone trop petite ↑ deadzone_px (10-15)

6. Recette de tuning pas à pas

Ki = 0, Kd = 0. On isole le proportionnel. Monte Kp progressivement (0.3 → 0.5 → 0.8 → 1.0) jusqu'à ce que la tête suive la cible mais commence à osciller. Note cette valeur — c'est le Kp critique. Prends 60-70% de cette valeur.

Monte Kd depuis 0 par pas de 0.01 (0.02 → 0.04 → 0.06 → 0.08). L'oscillation doit disparaître. Arrête quand la tête se stabilise sans devenir trop lente. Vise un ratio Kd/Kp entre 0.08 et 0.15.

Ajoute Ki uniquement si tu vois un décalage permanent (la tête s'arrête un peu à côté de la cible). Commence à 0.01, monte par pas de 0.005. En général 0.01 à 0.03 suffit. Si ça re-oscille, baisse.

Ajuste vel_gain si le mouvement est bon mais trop lent globalement. Monte de 4.0 vers 6.0 ou 8.0. C'est indépendant du PID — ça contrôle juste le Moving_Speed envoyé au servo.

Ajuste smoothing_alpha si les transitions sont trop brusques (monte vers 0.15) ou trop molles (monte vers 0.35). Et deadzone_px si la tête tremble quand la cible est centrée (10-15 px).

7. Commandes ROS2 en live

Tout se modifie à chaud sans relancer le node :

# PID
ros2 param set /head_tracker_node pid_pan_kp 0.6
ros2 param set /head_tracker_node pid_pan_ki 0.02
ros2 param set /head_tracker_node pid_pan_kd 0.08

# Idem pour tilt
ros2 param set /head_tracker_node pid_tilt_kp 0.6
ros2 param set /head_tracker_node pid_tilt_ki 0.02
ros2 param set /head_tracker_node pid_tilt_kd 0.08

# Vitesse
ros2 param set /head_tracker_node vel_gain 4.0
ros2 param set /head_tracker_node min_vel 0.3

# Lissage
ros2 param set /head_tracker_node smoothing_alpha 0.20
ros2 param set /head_tracker_node deadzone_px 12.0

# Vérifier les valeurs courantes
ros2 param dump /head_tracker_node
Astuce : ouvre un terminal avec le log du head_tracker (ros2 topic echo /cmd_joints) pendant que tu modifies les params dans un second terminal. Tu verras les commandes changer en temps réel.

8. Valeurs recommandées

# head_tracker_params.yaml — valeurs anti-oscillation
head_tracker_node:
  ros__parameters:
    pid_pan_kp:  0.6
    pid_pan_ki:  0.02
    pid_pan_kd:  0.08

    pid_tilt_kp: 0.6
    pid_tilt_ki: 0.02
    pid_tilt_kd: 0.08

    vel_gain: 4.0
    min_vel:  0.3
    max_pan_vel:  5.0
    max_tilt_vel: 3.0

    smoothing_alpha: 0.20
    cmd_threshold_rad: 0.005
    deadzone_px: 12.0
Pan et tilt séparés : en pratique le tilt (vertical) a souvent besoin de gains légèrement différents car la charge mécanique n'est pas la même. Si le pan est bon mais le tilt oscille encore, baisse pid_tilt_kp à 0.5 et monte pid_tilt_kd à 0.10.
